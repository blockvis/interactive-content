# Общие соглашения для всех слайд-промптов

> Этот файл — фундамент для каждого `slide-NN-*.md` в этой папке. Читается
> агентом **один раз** перед тем, как браться за конкретный слайд.
>
> Источник правды по сюжету: `../BRIEF.ru.md`.
> Источник правды по платформе: `../../../meta/MANIFEST.ru.md`.
> Источник правды по визуалу: `../../../presentation-classes/ratsif-2026-spring/class.md`.

## 1. Что такое «слайд-промпт»

Каждый файл `slide-NN-*.md` — это задание на визуальную разработку
одного слайда презентации. На выходе у этого задания:

1. **Markdown-файл слайда** в `presentations/gq128-beauty-presentation/ru/slideNNN.ru.md`
   (канонический русский; остальные 6 языков появятся через `npm run translate`).
2. **Ноль или более React-компонентов** в
   `presentations/gq128-beauty-presentation/components/<ComponentName>.tsx`
   и регистрация каждого в `components/index.tsx`.
3. **Ноль или более ассетов** в `presentations/gq128-beauty-presentation/assets/`.

Промпт описывает **что именно** и **как именно** должно получиться.
Он не должен оставлять открытых дизайн-вопросов — если вопрос остался, он
выносится в блок «Открытые вопросы» и блокирует работу до ответа автора.

## 2. Главный дидактический контракт (BRIEF §7 «кончик скотча»)

Повторяю сюда целиком, потому что без него любая реализация рискует
превратиться в лекцию:

1. **Никогда не начинать слайд с собственной абстракции.** Всегда — с боли
   инженера или общеизвестного факта, с которым зритель уже согласен.
2. **Только после того, как зритель признал проблему** своей, можно ввести
   один новый абстрактный объект — и ровно один за раз.
3. **Проверка.** Если убрать первые 10 секунд интерактива и показать слайд
   «из середины», должно остаться ощущение продолжающегося движения, а не
   лекции. Если нет — слайд не готов, откладываем.

## 3. Разделение Content / Backstage

Каждый слайд-файл имеет одну строку-маркер `<!-- backstage -->` в теле,
которая делит его на два таба (контракт платформы, MANIFEST §«Бэкстейдж
слайда»). Содержание этих табов подчиняется жёсткой политике плотности:

| | Content (сцена) | Backstage (wiki-страница) |
|---|---|---|
| Аудитория | все — от школьника до PhD | любопытные, самостоятельное чтение |
| Время потребления | 60–90 секунд живого доклада | минуты–часы, после/мимо доклада |
| Плотность | **нулевая**: одна картинка + одна строка | **высокая**: эссе, таблицы, формулы, ссылки |
| Текст | максимум **одна** строка-подпись | свободный markdown + KaTeX |
| Формул на сцене | **не больше одной**, и только если она сама — визуальный объект | без ограничений |
| Таблиц на сцене | **ноль**, единственное исключение — слайд 13 | сколько нужно |
| Интерактив | **обязателен** (кроме title и section) | опционален |

## 4. Контракт React-компонентов

### Имена и расположение

- PascalCase для компонента: `ComplexAction`, `GosperIsland`.
- Файл: `presentations/gq128-beauty-presentation/components/<PascalCase>.tsx`.
- В markdown директива пишется в kebab-case: `:::complex-action{...}`.
- Регистрация в `components/index.tsx`: ключ в `components` совпадает с
  kebab-case директивы.

### Контракт props

- **Все числовые props** допускают строку (remark-directive парсит атрибуты
  как строки). Компонент делает `Number(value)` или вынесенную
  утилиту-парсер.
- **Булевы атрибуты без значения** (`showGrid` без `=`) приходят как `true`.
- Все props **типизированы** TypeScript-интерфейсом.
- У каждого компонента есть **дефолты** для всех необязательных props —
  презентация должна собираться, даже если автор забыл атрибут.

### Интерактив и анимация

- Анимация — `requestAnimationFrame` или `framer-motion` (разрешено то и
  другое, но один компонент — одна технология).
- SVG — предпочтительный рендер для всего, что геометрично и не требует
  3D. Canvas / WebGL — только если SVG реально не тянет (облако 10 000+
  частиц, 3D-рендер, кривые очень высокой глубины).
- Перетаскиваемые ручки — через pointer events (поддержка мыши, тача,
  стилуса в одном коде).
- **Запрет** на анимации, которые играют «сами по себе» без жеста
  пользователя, длиннее 3 секунд. Любое автоматическое движение, которое
  длится дольше, требует паузы и возобновления по клику — иначе оно крадёт
  внимание от голоса докладчика.

### Reader mode / responsive

Платформа оборачивает слайд в контейнер, который может быть **широким**
(presentation mode, ≥ 769 px) или **узким** (reader mode, ≤ 768 px).
Компонент **обязан** корректно работать в обоих режимах:

- Всё, что в presentation mode — горизонтальная лента из двух панелей,
  в reader mode складывается в вертикаль.
- Текст подписей не мельчает ниже 14 px.
- Жесты, требующие мыши (hover), дублируются тапом.
- SVG-сцены используют `viewBox` и масштабируются без потери качества.

### Reduced motion

`@media (prefers-reduced-motion: reduce)` — компонент **обязан** либо
заменять анимацию статическим финальным кадром, либо снижать её до едва
заметной. Никаких пульсаций, параллаксов, стробоскопов по умолчанию.

### Accessibility

- Роли ARIA там, где это осмысленно (`role="img"` + `aria-label` для
  чисто-визуальных сцен).
- Интерактивные ручки — `role="slider"` + `aria-valuenow` / `valuemin` /
  `valuemax`, реагируют на стрелки клавиатуры.
- Цвет никогда не единственный носитель смысла: у цветных элементов есть
  форма или подпись-дубликат.
- Контраст текста к фону — минимум WCAG AA (4.5:1 для основного текста).

## 5. Цветовые токены

### Токены класса `ratsif-2026-spring` (из `class.md`)

CSS-переменные доступны в любом компоненте через `var(--class-*)`:

| Токен | Значение | Назначение |
|---|---|---|
| `--class-background` | `#FFFFFF` | фон |
| `--class-foreground` | `#000000` | основной текст |
| `--class-primary` | `#344068` | заголовки, рамки, primary-элементы |
| `--class-accent` | `#1CADE4` | хайлайты, активное состояние |
| `--class-muted` | `#D9E0E6` | сабфоны, неактивное |

Бренд-полосы:

- `--class-stripe-dark` = `#214673` (TSI navy)
- `--class-stripe-light` = `#2683C6` (Retrospect accent2)

### Цветовая кодировка цифр Эйзенштейна (BRIEF §6.5)

Эти токены **пока не зафиксированы в `class.md`** — ждут подтверждения
автора. Однако они используются на слайдах 6, 10, 11, 12, 14 как общий
визуальный язык. Рекомендованная форма инъекции:

```css
:root {
  --digit-zero:           transparent;    /* 0,        alpha-only */
  --digit-plus-one:       #00A651;        /* +1,       12h, green */
  --digit-plus-omega:     #0072CE;        /* +ω,       4h,  blue */
  --digit-plus-omega-inv: #E4002B;        /* +ω⁻¹,     8h,  red */
  --digit-minus-one:      #D6007F;        /* −1,       6h,  magenta */
  --digit-minus-omega:    #FFD100;        /* −ω,       10h, yellow */
  --digit-minus-omega-inv:#00B3BE;        /* −ω⁻¹,     2h,  cyan */
}
```

Конкретные hex’ы — примерные «чистые» RGB/CMY, финальные значения утвердит
автор. Ключевое: **green/blue/red — это RGB-primaries**, они садятся на
$\{+1, +\omega, +\omega^{-1}\}$; **magenta/yellow/cyan — CMY** на
$\{-1, -\omega, -\omega^{-1}\}$; они попарно дополнительны.

Каждое промптовое задание, использующее эти цвета, **обязано** явно
упомянуть, что оно зависит от этих токенов, и при отсутствии токенов —
инлайнить их локально в компоненте, помечая `// TODO: вынести в class.md`.

## 6. Типографика

Из `class.md`:

- Heading: Open Sans, 300 (light) для заголовков, 600/700 — акценты.
- Body: Open Sans, 400; 500 — второстепенные акценты.
- Базовый кегль: 20 px.
- Heading scale: 1.333.
- Line-height: 1.45.

Подпись на сцене (строка под интерактивом) — `text-xl` в Tailwind (≈ 20 px)
или `1em`, **без жирного**, курсивом разрешено подчёркивать авторские
формулировки из BRIEF §14.

## 7. Ассеты

- Путь в слайде — `../assets/<file>`. Компилятор превращает в веб-URL.
- Изображения — предпочтительно SVG; для фото и сканов — WEBP, максимальный
  размер 2000×2000 px.
- Исторические сканы и цитаты — в подпапку `../assets/historical/`.
- Иконки — inline SVG внутри компонентов, не отдельные файлы.

## 8. Структура markdown-слайда

Стандартный каркас, к которому приводится каждый слайд:

```markdown
---
layout: content
title: "Один из трёх: title | section | content"
subtitle: "Опционально; появляется подписью над title"
---

Одна строка-подпись (или прямая остановка, если есть только интерактив).

:::component-name{prop=1 another=two}
Подпись к интерактиву (необязательно).
:::

<!-- backstage -->

## Заголовок блока

Свободный markdown + KaTeX: `$...$` inline, `$$...$$` block.

...дальше разделы бэкстейджа, столько, сколько нужно...
```

Frontmatter:

- `layout: title` — только для слайда 1 (big QR, нет таба Content/Backstage).
- `layout: section` — для слайдов 2 и 15 (заставочные).
- `layout: content` — для всех остальных (слайды 3–14).
- `title` / `subtitle` — опциональные.
- `qr: false` — escape hatch; использовать только если интерактив прямо
  конфликтует с углом, где рендерится QR.

## 9. Контракт приёмки слайда

Слайд считается готовым, если:

1. Файл `slideNNN.ru.md` собирается без ошибок (`npm run compile`).
2. Компонент (если есть) **работает в presentation mode** на десктопе и
   **в reader mode** на мобильной ширине.
3. `prefers-reduced-motion: reduce` не ломает слайд.
4. Контраст текста проходит WCAG AA.
5. Жесты дублируются клавиатурой (Tab + Enter / стрелки).
6. `<!-- backstage -->` делит слайд на content и backstage так, как описано
   в конкретном промпте.
7. Backstage **плотен и верен** (ссылки на BRIEF + собственные факты);
   content **лёгок и не содержит формул больше одной**.
8. Дидактический контракт §2 проходит проверку «слайд с середины».

## 10. Что запрещено во всех слайдах

- **Жаргон на сцене.** «Эрмитовость», «коммутатор», «расщепление в
  $\mathbb Z[\omega]$» — только в бэкстейдже.
- **Внешние зависимости** (three.js, d3-full, анимационные библиотеки)
  **без согласования.** Разрешены: `framer-motion`, `katex` (уже в
  пайплайне). Всё новое — явно обосновать в промпте конкретного слайда.
- **Запросы к сети на рендере.** Слайд — offline-first.
- **Любое цитирование HTML- или PDF-файлов из `raw-ai-articles-dirty/`**
  (BRIEF §6): это частные заметки, не публикуемый источник. Факты оттуда
  переформулировать своими словами.
- **Автозапуск `npm run translate`** — только по прямой команде автора.

## 11. Типичная форма слайд-промпта

Каждый файл `slide-NN-*.md` следует этой структуре:

```
# Slide NN — <краткое имя>

## Роль в арке доклада
## Длительность и «кончик скотча»
## Content — визуал, интерактив, подпись
## Backstage — разделы, плотные факты, ссылки
## Компоненты для реализации (если есть)
  Для каждого: имя, путь, props, state, анимация, reader mode,
  a11y, reduced motion.
## Ассеты (если нужны)
## Токены и стили
## Проверка «кончик скотча» и «слайд с середины»
## Что НЕ делать
## Открытые вопросы (если остались)
```
