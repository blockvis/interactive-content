# Локальная разработка и её фен-шуй

Короткая памятка о том, как устроен `npm run dev` и как с ним жить,
особенно во время длинных AI-рефакторингов, когда по коду прокатывается
волна правок.

## Одна команда на всё

```
npm run dev
```

Под капотом — `node scripts/dev.mjs`, который делает сразу три вещи:

1. **Компиляция презентации** (`scripts/compile.mjs`). Читает
   `presentations/<slug>/presentation.md`, резолвит класс через
   `class.md`, пишет `src/generated/slides.json` **атомарной записью**
   (`tmp` → `rename`), чтобы Turbopack не успел прочитать полу-файл.
2. **Запуск `next dev`** (Turbopack).
3. **Watcher** на папке презентации: при изменении `*.md`, `assets/`
   или `components/` пере-компилирует JSON. Next подхватывает новый
   JSON своим HMR.

Другую презентацию можно указать аргументом:
`npm run dev -- path/to/other-presentation`.

## Почему localhost раньше «ложился» и не вставал

Исторически `scripts/dev.mjs` делал `next.on('exit') → process.exit`:
если Next по какой-то причине падал (панико́вал Turbopack на
транзиторно-невалидном TS/JSX в середине AI-рефакторинга, OOM, segfault),
то родительский скрипт выходил следом. Ничто не поднимало сервер
обратно — приходилось перезапускать руками.

## Самовосстановление

Теперь в `dev.mjs` живёт супервизор:

- Если `next dev` завершился **не по Ctrl+C** — скрипт освобождает
  порт 3000 (`lsof -tiTCP:3000` + kill залипших процессов) и
  перезапускает Next с экспоненциальным бэкоффом: 300мс → 600 → 1.2с →
  2.4с → максимум 5с. После 10 секунд стабильной работы счётчик
  сбрасывается.
- Перед каждым запуском — короткое ожидание свободного порта
  (на случай TIME_WAIT).
- `SIGINT`/`SIGTERM` (Ctrl+C, `kill`) — честный выход, супервизор не
  пытается реанимировать.
- Стартовая компиляция **не фатальна**: если в момент запуска
  манифест презентации временно битый, `dev` всё равно поднимет
  Next и ждёт, пока watcher пере-компилит валидный JSON.

## Фен-шуй: что делать во время больших AI-рефакторингов

- **Не перезапускать руками.** Если Turbopack упал посреди волны
  правок — супервизор поднимет Next сам за ≤ 5 секунд после того, как
  файлы перестанут меняться. Подожди 2–5 секунд и обнови страницу.
- **Не закрывать терминал с `npm run dev`.** Он — твой контроллер
  перезапусков.
- **Если порт 3000 занят «извне»** (другой процесс, не Next) —
  супервизор всё равно попытается его освободить `kill -9`, но
  честнее остановить конкурента самому.
- **Если супервизор крутит бэкофф по кругу** (видно по сообщениям
  `next dev exited … Restarting in …`) — значит конфигурация
  действительно сломана (синтаксическая ошибка в `next.config.*`,
  битый TS-корень, отсутствующий `node_modules`). Читай вывод
  Next — там будет реальная причина. Супервизор тут не поможет.
- **Презентация временно в невалидном состоянии** (AI перезаписал
  frontmatter и ещё не закончил) — это **нормально**. `compile.mjs`
  напечатает ошибку, но сервер останется жив и восстановится при
  следующем валидном сохранении.

## Когда этого мало

Текущий супервизор закрывает кейс «Next упал — подними обратно».
Он **не** закрывает:

- Next формально живой, но завис (не отвечает на HTTP). Решение
  на будущее — health-probe: периодический `fetch('/')`, три
  фейла подряд → SIGTERM, супервизор перезапустит. Добавим, если
  начнёт реально мешать.
- `fs.watch({recursive: true})` на macOS иногда молча перестаёт
  слать события под шквал атомарных переименований. Если заметишь,
  что watcher «оглох» — перезапусти `npm run dev`. Устойчивый
  watcher (recreate-on-error) — тоже кандидат в пункт 3 плана.
