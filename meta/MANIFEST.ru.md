# Meta-Meta

Этот файл — **каноническая** версия манифеста проекта. Канонический язык — русский.

Манифест описывает **только актуальное состояние** продукта. Отброшенные подходы, предыдущие ревизии формата и путь «как мы к этому пришли» здесь не фиксируются: проект в режиме полёта (до релиза), потребителей у него ещё нет, и историю вести не нужно — она лишь засоряет чтение. После первого релиза появится отдельное версионирование форматов и их история миграций; до тех пор единственная обязанность манифеста — точно описывать **то, что есть сейчас**.

При обсуждениях в чате важные архитектурные и полит-решения агент проактивно переносит сюда (см. правило `.cursor/rules/manifest-workflow.mdc`).

# Что это такое

Репозиторий-контейнер для интерактивных презентаций. Каждая презентация — отдельная папка в `presentations/<slug>/`, где `<slug>` одновременно является путём презентации на домене. Репозиторий принимает пулл-реквесты с новыми презентациями.

# Структура контента

Проект оперирует двумя параллельными каталогами:

- `presentations/<slug>/` — **сами презентации**.
- `presentation-classes/<slug>/` — **классы презентаций**: стили, темы, chrome конференции. Один класс обслуживает все презентации одного мероприятия / серии и гарантирует единый визуальный язык.

## Мета-файлы

В каждом каталоге лежит Markdown с YAML-frontmatter:

- `presentations/<slug>/presentation.md` — мета конкретной презентации: ссылка на `class`, список `languages` (первый — канонический), заголовок, авторы, контекст мероприятия. Тело — описание.
- `presentation-classes/<slug>/class.md` — мета класса: имя, данные мероприятия, дизайн-токены (палитра, шрифты, chrome), визуал обязательных платформенных фич. Тело — описание.

Формат — тот же, что и у слайдов (MD + frontmatter + KaTeX + относительные ассеты): единый парсер (`gray-matter`), единый ментальный контракт, YAML допускает комментарии, тело остаётся живым описанием и читаемо прямо на GitHub.

## Версионирование формата

Каждый мета-файл (и `presentation.md`, и `class.md`) объявляет во frontmatter поле `spec: <version>`. Сейчас `spec: 0.1` — стартовая ревизия формата перед релизом.

До первого релиза политики обратной совместимости нет: `spec` инкрементится свободно вместе с форматом по мере необходимости. После релиза вокруг этого поля появится миграционный слой и правила эволюции — тогда же манифест начнёт вести историю изменений формата.

# Три уровня контракта

Ответственность распределена между тремя уровнями. Их нельзя смешивать: каждое обязательство «приписано» ровно к одному уровню.

## 1. Платформа (инвариант)

Движок гарантирует на любой презентации, независимо от класса и содержимого:

- **Responsive layout.** Слайд — это HTML-страница, а не жёсткий canvas. На проекторе заполняет 16:9 естественно, на мобильном гармонично переформатируется.
- **Sticky footer-навигация.** Присутствует на каждом слайде. Платформа строит `nav`-слот, класс **обязан** разместить его в каждом layout-е; оформление — за классом, отказ от рендеринга nav нарушает контракт.
- **Переключатель языка** (в chrome).
- **QR на титульном слайде** (в presentation mode) — указывает на URL презентации. Факт наличия — инвариант; размер, позиция и оформление — целиком за классом.
- **Слот для QR текущего слайда** (в presentation mode) — платформа умеет его рендерить, но он **активируется и оформляется классом** (`chrome.qrPerSlide`). Если класс молчит — слота нет.
- **Два канонических таба слайда** — `content` и `backstage`. Платформа пре-строит слоты `tabSwitcher` (переключатель; `null` если у слайда нет бэкстейджа) и `slideBody` (markdown активного таба; всегда не-null). Состояние живёт на клиенте, синхронизируется с `location.hash` (`#backstage`). Класс обязан разместить `slideBody` и волен решать, куда класть `tabSwitcher` (обычно — в sticky-шапку над основной колонкой).
- **Режимы рендеринга:**
  - **Presentation mode** — широкий вьюпорт (проектор, десктоп): chrome и QR активны.
  - **Reader mode** — мобильный / узкий вьюпорт (`max-width: 768px` или близко): QR скрыты, sticky nav сохраняется.
  - Автоопределение по вьюпорту, override через query-параметр `?mode=presentation` / `?mode=reader`.

## 2. Класс (визуал + layouts)

Класс — не просто пакет токенов, а **React-модуль**, который провидит layouts для каждой роли слайда. По структуре близко к PowerPoint slide master / slide layout: платформа собирает страницу, но **как** она рисуется — решает класс.

```
presentation-classes/<slug>/
  class.md             ← токены + метадата (YAML frontmatter + описание)
  index.tsx            ← ClassModule: layouts, components
  layouts/
    title.tsx          ← LayoutComponent для slide.layout = "title"
    content.tsx        ← для "content" (и обычно для "section")
    section.tsx        ← опционально
  chrome/              ← классовые chrome-элементы (полосы, логотип, фон)
  components/          ← директив-компоненты, используемые в markdown
  lib/                 ← внутренняя кухня класса (форматтеры, хелперы)
  source/              ← первоисточник (pptx / brandbook)
  assets/              ← веб-готовые ассеты мероприятия
```

Контракт класса описан в `src/lib/class-module.ts`:

```ts
export interface ClassModule {
  layouts: Partial<Record<LayoutName, LayoutComponent>>;
  components?: Record<string, ComponentType<Record<string, unknown>>>;
}

export interface LayoutProps {
  slide: Slide;
  presentation: PresentationMeta;
  classMeta: ClassMeta | null;
  platform: PlatformSlots; // { nav, languageSwitcher, titleQr, cornerQr, lang, ... }
}
```

Платформа пре-строит slot-ы (sticky nav, language switcher, QR) и передаёт их layout-у; layout решает, куда их класть. Инварианты (факт наличия обязательных фич) остаются за платформой, визуал и вёрстка — за классом.

### Дефолтный класс `example`

`presentation-classes/example/` — минимальный корректный `ClassModule`. Это одновременно:

- **Fallback**: если презентация не объявляет `class:` или объявленный класс не имеет `index.tsx`, `compile.mjs` переключается на `example`, сборка не падает.
- **Reference implementation**: от `example/` удобно копировать структуру при написании нового класса.

### Активный класс подключается через сгенерированный файл

`scripts/compile.mjs` читает `presentation.md.class`, проверяет наличие `index.tsx` у нужного класса и генерирует `src/generated/class.tsx`, который реэкспортирует активный `ClassModule`. Платформа импортирует `@/generated/class` и не знает про конкретный slug — только про интерфейс.

### Evolution, not revolution

Класс существует, чтобы наша веб-презентация **гармонировала** с соседним pptx-докладом на том же проекторе. Мы не изобретаем визуальный язык — мы его цитируем. Когда у конференции есть официальный шаблон (pptx / brandbook), он кладётся в `presentation-classes/<slug>/source/` и становится первоисточником правды, а `class.md` + layouts в `index.tsx` — его задокументированным портом на веб. Без шаблона — плейсхолдеры, явно помеченные как временные.

## 3. Слайд (контент)

`presentations/<slug>/<lang>/slide*.<lang>.md` — авторская свобода. Слайды лежат в подкаталогах по языкам; суффикс языка в имени файла сохраняется (избыточно с папкой, но даёт однозначный поиск по репо и страховку: компилятор предупреждает, если суффикс не совпадает с подкаталогом, и игнорирует файлы с маской `slide*.md` в корне презентации).

- Markdown + KaTeX (inline `$...$`, block `$$...$$`).
- Относительные ссылки на `assets/` презентации (изображения, видео, pdf, шрифты...).
- Встраиваемые React-компоненты из `components/` презентации.
- Frontmatter:
  - `title`, `subtitle`.
  - `layout: title | section | content` — роль слайда. По `layout: title` движок инжектирует большой QR.
  - `qr: false` — escape hatch, если `qrPerSlide` активен и мешает содержимому конкретного слайда.
- **Бэкстейдж слайда.** Тело после frontmatter может содержать одну строку-маркер `<!-- backstage -->` (HTML-комментарий) — она делит слайд на две канонические части: «Слайд» (то, что видит аудитория) и «За кадром» (расширенные материалы для самостоятельного изучения — метафора wiki-статьи и её страницы обсуждения). Имена вкладок в файле не указываются: они фиксированы платформой и локализуются через платформенный словарь (с оверрайдом класса через `chrome.tabs`). Если маркера нет — слайд одностраничный, переключатель не рендерится. Маркер невидим в любом markdown-рендере (GitHub в том числе), поэтому `---` остаётся свободным под обычные тематические разрывы внутри любой из частей.

# Встраивание компонентов

Синтаксис — `remark-directive`, чтобы остаться в пределах Markdown и сохранить читаемость на GitHub:

- Inline:  `::complex-plane{radius=1 showGrid}`
- Block:

  ```
  :::interactive-demo{initial=42}
  Подпись.
  :::
  ```

Резолв по цепочке (specificity высокая → низкая): `presentations/<slug>/components/` → `presentation-classes/<class>/components/` (из `ClassModule.components`) → платформенные defaults. Объединённый реестр собирается в `src/components/markdown-slide.tsx`.

На GitHub директивы отображаются как обычный текст — сознательная graceful degradation. Всё остальное (картинки по относительным путям, KaTeX-формулы, обычный markdown) GitHub рендерит сам, и это наш контракт: `.md`-файлы слайдов остаются человекочитаемым документом на GitHub без движка.

# Ассеты

Каталог — `presentations/<slug>/assets/`. Там лежат любые медиа-ресурсы слайдов: картинки, видео, pdf, шрифты, svg. Слайд лежит в `<lang>/`, поэтому канонический путь из слайда — `![alt](../assets/foo.png)`. Компилятор нормализует `../` и переписывает в веб-URL (`/content/<slug>/assets/foo.png`), копируя ассеты в `public/`. GitHub резолвит те же ссылки нативно, поэтому файлы остаются читабельными прямо на GitHub.

# Переводы

Канонический язык презентации — `languages[0]` из `presentation.md`; на нём пишет автор. Остальные языки (`languages[1..]`) — переводы, и они генерируются отдельной командой `npm run translate` ([scripts/translate.mjs](../scripts/translate.mjs)). Команда не встроена в dev-watcher намеренно: перевод — это деньги и ответственный акт, а не побочный эффект сохранения файла.

Инкрементальность держится на контент-хеше, а не на времени модификации (mtime сбрасывается любым `git clone`/rebase). Переведённый файл во frontmatter хранит два поля происхождения:

- `sourceHash: sha256-<hex64>` — хеш сырого содержимого канонического файла (frontmatter + тело) на момент перевода;
- `translatedBy: <model-id>` — id модели, сгенерировавшей этот перевод (напр. `claude-opus-4-7`). У ручных переводов поле отсутствует.

Небольшой информационный бейдж рядом с переключателем языка платформа показывает на **любом** неканоническом языке — факт того, что перед зрителем не оригинал, важнее, чем природа перевода. Визуально: цветной эмодзи `ℹ️`, в случае машинного перевода рядом с ним — лейбл `AI`. Полная информация (откуда переведено, какой моделью) — во всплывающей подсказке и в `aria-label`.

При повторном запуске скрипт сравнивает текущий хеш оригинала с записанным:

- совпадает → `skip` (перевод актуален);
- отличается → вызывает модель заново;
- у перевода нет `sourceHash` → `skip with warning`: считаем файл ручным переводом и не перетираем без явного `--force`.

Модель и ключ:

- `ANTHROPIC_API_KEY` — в `.env.local` (уже под `.env*` в `.gitignore`, не коммитится). npm-скрипт подхватывает через нативный Node-флаг `--env-file=.env.local`.
- `ANTHROPIC_MODEL` — опциональный override; дефолт захардкожен в скрипте.

Команды:

```
npm run translate                # все не-канонические языки × все устаревшие слайды
npm run translate -- en          # только язык `en`
npm run translate -- en 42       # только слайд #42 на `en`
npm run translate -- --force     # игнорировать sourceHash, переводить всё
npm run translate -- --dry       # показать план без обращений к API и записи файлов
```

Контракт постпроцессинга: после возврата от модели скрипт перезаписывает в переводе «структурные» поля (`layout`, `qr`, `id`, `class`, `spec`) значениями из оригинала — это защита от галлюцинаций. Переводятся только пользовательские строки: `title`, `subtitle`, `section` во frontmatter; проза, alt-тексты, подписи ссылок в теле; markdown-разметка, KaTeX, remark-директивы, код, HTML-комментарии (включая `<!-- backstage -->`), пути и URL остаются нетронутыми.

Сейчас переводятся только `slide*.<lang>.md`. `presentation.md` и `class.md` (title/subtitle/description/authors) локализуются вручную; команда для них появится, когда реально понадобится.
